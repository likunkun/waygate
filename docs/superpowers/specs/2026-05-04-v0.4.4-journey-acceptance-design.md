# V0.4.4 Journey Acceptance Layer Design

## Status

Draft for review. This is a planning artifact only; it does not implement V0.4.4.

## Goal

V0.4.4 adds a controller-enforced Journey Acceptance Layer so a workflow cannot be marked complete when local units pass but the end-to-end user journey is missing, unmapped, unverified, or failed.

The target guarantee is:

```text
Active Journey -> mapped AC -> mapped Unit/Test Case -> controller-run command -> passed journey evidence -> Final Acceptance
```

## Non-Goals

- Do not add an independent `approvals/journeys.md` gate in V0.4.4.
- Do not migrate Requirements, AC, Unit Plan, or Journey contracts to full V0.7 structured facts.
- Do not implement V0.4.5 Final Scope Audit.
- Do not implement V0.4.6 strict "every non-manual AC must have a test case" enforcement.
- Do not claim V0.5 anti-forgery isolation, clean checkout verification, or role file policy.

## Threat Model

V0.4.4 is meant to stop lazy or incomplete agent behavior:

- no declared journey
- no journey-to-test mapping
- no executable closure/E2E command
- natural-language "verified" claims without command evidence
- unit evidence passing while journey evidence is missing

V0.4.4 only partially mitigates malicious forgery. If the same agent can freely edit state, approved gates, source code, and verifier artifacts, it can still forge data. Stronger protection belongs to V0.5 role file policy, verifier isolation, and clean checkout verification.

V0.4.4 must still reduce the forgery surface:

- Journey evidence is generated by controller verifier code from command results, not accepted from agent self-report.
- Approved Requirements / Unit Plan hashes are re-read before final acceptance.
- Journey contract changes must route through requirements revision and `change_requests.jsonl`.
- Final Acceptance revalidates artifacts instead of trusting state booleans alone.

## User-Facing Behavior

Requirements Gate remains the place where human reviewers confirm journeys. If a task describes a cross-step or closure user flow, the Requirements gate must expose that journey with stable identity and linked ACs.

Unit Plan Gate remains the place where human reviewers confirm execution coverage. Every active journey must map to at least one unit and at least one executable closure/E2E test case.

Verifier remains the only authority for pass/fail evidence. It runs commands from the approved Unit Plan and writes journey evidence rows derived from actual command results.

Final Acceptance Gate renders a Journey Matrix. A human can still reject and route feedback, but approval cannot advance to `DONE` if active journey evidence is missing or failed.

## Artifacts

V0.4.4 introduces:

```text
.plan-ralph/artifacts/journeys/
  journeys.json
  journey-evidence.json
```

`journeys.json` is the controller-owned journey contract extracted from approved Requirements and enriched during Unit Plan validation.

`journey-evidence.json` is the controller-owned summary of verifier-derived journey evidence across relevant unit artifacts.

Final Acceptance continues to render:

```text
.plan-ralph/approvals/final-acceptance.md
```

with an added `Journey Matrix` section.

## Journey Contract Schema

`journeys.json` should contain:

```json
{
  "version": 1,
  "generated_at": "2026-05-04T00:00:00+00:00",
  "source_gate": "requirements",
  "requirements_gate_path": ".plan-ralph/approvals/requirements-and-acceptance.md",
  "requirements_gate_hash": "sha256-body-hash",
  "unit_plan_gate_path": ".plan-ralph/approvals/unit-plan.md",
  "unit_plan_gate_hash": "sha256-body-hash-or-null",
  "journeys": [
    {
      "journey_id": "J-001",
      "title": "User retries an offline import",
      "status": "active",
      "steps": [
        "User starts an import with a temporary network failure",
        "User retries from the error state",
        "System completes import without duplicate records"
      ],
      "linked_acceptance_criteria": ["AC-07"],
      "linked_units": ["unit-import-retry"],
      "verification_layer": "e2e",
      "verification_command": "pytest tests/e2e/test_import_retry.py -q",
      "test_cases": ["TC-AC-07-J-001"]
    }
  ]
}
```

Minimum required fields per active journey:

- `journey_id`
- `title`
- `status`
- `steps`
- `linked_acceptance_criteria`
- `verification_layer`

Unit Plan approval must enrich or validate:

- `linked_units`
- `verification_command`
- `test_cases`

## Evidence Schema

Verifier should derive `journey_evidence_rows` from approved test cases and actual command results:

```json
{
  "journey_id": "J-001",
  "title": "User retries an offline import",
  "acceptance_criteria": ["AC-07"],
  "unit_id": "unit-import-retry",
  "test_case_id": "TC-AC-07-J-001",
  "layer": "e2e",
  "command": "pytest tests/e2e/test_import_retry.py -q",
  "status": "passed",
  "returncode": 0,
  "expected": "Import completes and no duplicate records are created",
  "artifact_refs": [
    ".plan-ralph/artifacts/unit-import-retry/verification.json"
  ]
}
```

Allowed statuses:

- `passed`
- `failed`
- `missing`

Only `passed` satisfies Final Acceptance.

## Enforcement Points

### Requirements Approval

Controller validates the Requirements gate before approval.

If requirements contain cross-step user flows, closure language, E2E verification needs, or explicit user journeys, the controller requires journey rows or parseable journey sections.

Failure blocks `approve --gate requirements` with a controller validation error.

The controller writes or refreshes:

```text
.plan-ralph/artifacts/journeys/journeys.json
```

### Unit Plan Approval

Controller validates that every active journey maps to execution:

- at least one linked unit
- at least one linked test case
- executable command present in the unit's `verification_commands`
- concrete fixture/setup and expected assertion
- `layer=e2e` or unit `workflow_validation_level=closure` for closure journeys

Failure blocks `approve --gate unit-plan`.

### Verifier

Controller-run verifier derives `journey_evidence_rows` from:

- active journeys
- current unit test cases
- command execution results
- existing verifier artifact refs

Agent-written journey pass/fail claims are not authoritative.

### Final Acceptance

Controller renders a Journey Matrix and validates again on approval.

Approval is blocked if:

- `journeys.json` is missing for a task that requires journeys
- an active journey has no passed evidence row
- a journey maps to a failed/missing command result
- approved Requirements or Unit Plan gate hashes no longer match the contract inputs

## Change Requests

Any requirements revision or Final Acceptance `requirements` route that changes active journeys must append `change_requests.jsonl`.

The existing `impacted.journeys` field should remain the audit hook. V0.4.4 does not need a separate change request type.

## Final Acceptance Rendering

Add a section:

```markdown
## Journey Matrix

| Journey | AC | Unit | Test Case | Layer | Status | Command / Evidence | Expected | Artifacts |
| --- | --- | --- | --- | --- | --- | --- | --- | --- |
| J-001 User retries an offline import | AC-07 | unit-import-retry | TC-AC-07-J-001 | e2e | passed | `pytest tests/e2e/test_import_retry.py -q` | no duplicates | artifacts/unit-import-retry/verification.json |
```

The matrix is evidence for human review, but the hard gate is the controller validator.

## Implementation Units

### Unit 1: Journey Contract Artifact

Create journey extraction and contract writing under `artifacts/journeys/journeys.json`.

Validation level: `fragment`

Done when Requirements approval blocks malformed journey contracts and accepts valid contracts.

### Unit 2: Unit Plan Journey Mapping Validator

Validate that active journeys map to executable closure/E2E test cases.

Validation level: `fragment`

Done when Unit Plan approval rejects missing journey mapping and accepts valid mapping.

### Unit 3: Verifier Journey Evidence

Generate `journey_evidence_rows` from command results and mapped test cases.

Validation level: `fragment`

Done when passed, failed, and missing command outcomes produce correct journey evidence.

### Unit 4: Final Acceptance Journey Matrix Hard Gate

Render Journey Matrix and block final approval when active journey evidence is missing or failed.

Validation level: `closure`

Done when final approval cannot reach `DONE` without passed journey evidence.

### Unit 5: Docs and Regression

Update README and run full regression.

Validation level: `closure`

Done when docs describe actual V0.4.4 behavior and `workflow_controller/tests` passes.

## Open Questions

1. Should journey extraction require a strict Markdown table in Requirements Gate, or accept both a table and bullet sections for compatibility?
2. Should tasks with no explicit user journeys but with `verification: e2e` AC automatically require one synthetic journey?
3. Should `journey_evidence_rows` live only in unit `verification.json`, or also be aggregated into `artifacts/journeys/journey-evidence.json` after every verifier run?

Recommended answers:

1. Require a strict table for new V0.4.4 contracts, but tolerate existing bullet `User Journeys` as draft input before approval.
2. Yes. Any `verification: e2e` AC should require an active journey.
3. Do both. Unit artifacts preserve local evidence; aggregate artifact simplifies Final Acceptance.
